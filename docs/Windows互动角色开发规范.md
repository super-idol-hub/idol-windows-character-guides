# Windows 互动角色开发规范

版本：1.0

适用对象：AI 编程代理、角色项目维护者、素材制作者和发布验收人员

参考产品：[xiaoxiwei](https://github.com/super-idol-hub/xiaoxiwei)

## 1. 目标与执行原则

本规范用于从人物资料直接开发一款可发布的 Windows 互动角色。交付物必须是可运行、可测试、可分发的正式软件，不是网页原型、静态立绘播放器或只有部分动作的演示。

AI 执行时遵守以下原则：

1. 先完整阅读本规范，再修改代码或批量生成素材。
2. 只向用户确认会改变人物身份、视觉方案、授权边界或产品范围的信息。
3. 技术实现以本规范为默认答案，不为同一问题反复创建试验版本。
4. 先批准人物身份锚点和动作联系表，再批量生成 176 帧，避免后期整体返工。
5. 修复应针对失败的动作行、皮肤或测试项，不重新制造整套过渡版本。
6. 未生成最终 EXE、未运行 QA、未形成发布包时，不得声称项目已经完成。

本规范不绑定任何旧提交、旧目录或历史版本。`xiaoxiwei` 仅作为最终产品形态和工程思路的参考。

## 2. 开始前的最少输入

AI 开始前应获得或明确记录：

| 输入 | 要求 |
|---|---|
| 角色名称 | 用户界面和发布页显示的正式名称 |
| 项目标识 | 小写英文、数字和连字符组成，例如 `xiaoxiwei` |
| 人物资料 | 正面、不同角度、表情、全身、服装细节；不足时明确缺口 |
| 官方皮肤 | 首发必须内置的服装或造型列表，至少 1 套 |
| 目标仓库 | 新仓库或现有仓库及默认分支 |
| 使用边界 | 非官方/非商业默认值，或用户提供的明确授权 |

如果输入资料不足以稳定人物身份，AI 应停在“身份锚点确认”之前说明缺失内容；不得用猜测的人脸或服装直接完成整批资源。

## 3. 正式交付物

项目完成时至少包含：

- 一个可独立运行的 Windows EXE；
- 一个包含 EXE、使用说明和必要文件的完整 ZIP；
- 全部官方皮肤及其完整动作资源；
- 源码、构建脚本、应用清单和资源清单；
- 自动化自检与窗口 QA 的通过记录；
- 人工视觉验收记录；
- SHA-256 校验文件；
- 版本说明、权利说明和下载入口。

推荐仓库结构：

```text
.
├─ README.md
├─ CHANGELOG.md
├─ RELEASE_NOTES.md
├─ PROJECT-MANIFEST.json
├─ docs/
│  ├─ USER_GUIDE.md
│  ├─ RIGHTS.md
│  └─ QA_REPORT.md
├─ qa/
│  └─ evidence/
├─ skin-pack-template/
└─ source/
   └─ standalone/
      └─ <character-slug>/
         ├─ Program.cs
         ├─ app.manifest
         ├─ build.bat
         ├─ resources/
         └─ README.md
```

不要沿用参考角色的名字作为新项目目录。程序集名、窗口标题、托盘提示、注册表键、菜单文字、资源命名和发布文件名都必须使用当前角色信息。

## 4. 产品与运行架构

### 4.1 Windows 应用

默认实现为 C# 5、经典 .NET Framework、WinForms 分层透明窗口：

- 入口标记 `[STAThread]`；
- 无边框，不显示在任务栏，窗口置顶；
- 使用 layered window 与 tool window；
- 启动时位于主屏幕工作区右下方；
- 支持多显示器和 Per-Monitor DPI；
- 托盘图标提供显示、暂停、设置和退出；
- `Ctrl+Q` 可退出；
- 普通启动保持单实例；
- 自检和 QA 命令允许独立执行。

窗口默认缩放为 225%，允许范围 125%–400%，鼠标滚轮每次调整 25%。窗口缩放后应保持角色脚底锚点稳定，不突然跳位。

### 4.2 构建

默认编译器：

```text
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\csc.exe
```

建议参数：

```text
/target:winexe /optimize+ /debug- /platform:anycpu /langversion:5 /codepage:65001
```

引用：

```text
System.dll
System.Core.dll
System.Drawing.dll
System.IO.Compression.dll
System.Windows.Forms.dll
```

应用清单至少声明：

- `asInvoker`；
- `uiAccess="false"`；
- Common Controls v6；
- `PerMonitorV2`，并兼容 `PerMonitor`。

最终 EXE 必须内置所有官方皮肤、动作帧和运行所需资源。在干净的 Windows 环境中运行时，不得依赖 Python、Codex、OpenCV、开发工具、源码目录或项目专用 DLL。

### 4.3 菜单与设置

右键菜单至少包含：

- 动作；
- 皮肤；
- 暂停/继续；
- 始终置顶；
- 开机启动；
- 帮助/关于；
- 退出。

开机启动默认关闭，由用户主动开启。使用当前用户的 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`，不请求管理员权限。测试不得更改用户真实注册表设置。

## 5. 人物视觉与素材生产

### 5.1 身份锚点

批量生产前，先制作一张“身份锚点”：

- 正面或最容易识别人物的三分之二角度；
- 完整呈现脸型、五官、发型、身材比例和主服装；
- 采用最终产品的绘制风格；
- 透明背景，全身入画；
- 不包含文字、水印、背景、地面阴影或无关道具。

身份锚点必须由用户明确确认。之后所有皮肤和动作均以它为人物一致性基准。

### 5.2 动作联系表

在生成正式帧前，每个动作行先制作一张联系表，展示关键姿势和时间顺序。检查：

- 是否仍能一眼识别为同一人物；
- 动作是否符合人物特点，而不是性别刻板印象；
- 头身比、脸、发型和服装是否稳定；
- 手脚、道具和表情是否完整；
- 帧间变化能否形成自然动作；
- 角色脚底、身体大小和画面位置是否稳定。

联系表通过后才生成该行全部帧。

### 5.3 最终帧

每张动作帧必须：

- 528 × 808 像素；
- RGBA PNG；
- 全身完整，透明背景；
- 无文字、字幕、水印、场景和地面阴影；
- 人脸、体型、发型、服装和配色稳定；
- 同一动作行使用统一视口、缩放和脚底注册点；
- 特效与身体形成整体，不出现悬空碎片；
- 透明区域无脏边和大面积半透明残影。

所有官方皮肤必须拥有相同的动作协议，不得只为默认皮肤补齐动作。

## 6. 24 行、176 帧动作协议

资源目录按 `r00`–`r23` 分行、`c00`–`c07` 分帧。各行动画帧数为：

```text
7, 8, 8, 4, 5, 8, 6, 6, 6, 8, 8, 6,
8, 8, 8, 8, 8, 8, 8, 8, 8, 8, 8, 8
```

合计 176 帧。

| 行 | 帧数 | 正式用途 | 触发与约束 |
|---|---:|---|---|
| `r00` | 7 | 待机 | `c00`–`c05` 循环，`c06` 为中性回归帧 |
| `r01` | 8 | 向右移动兼容位 | 默认不触发，不允许自动漫游 |
| `r02` | 8 | 向左移动兼容位 | 默认不触发，不允许自动漫游 |
| `r03` | 4 | 挥手 | 双击或菜单触发 |
| `r04` | 5 | 跳起/受惊 | 拖动开始使用 `c00`–`c02`，按住时停在 `c02` |
| `r05` | 8 | 疲惫/失败 | 菜单或情境触发 |
| `r06` | 6 | 等待 | 一次性动作 |
| `r07` | 6 | 工作 | 一次性动作 |
| `r08` | 6 | 查看/检查 | 一次性动作 |
| `r09` | 8 | 凝视方向 0°–157.5° | 8 个直接方向帧，不按序播放 |
| `r10` | 8 | 凝视方向 180°–337.5° | 8 个直接方向帧，不按序播放 |
| `r11` | 6 | 恢复/皱眉/跺脚 | 拖动释放使用 `c02`–`c05` |
| `r12` | 8 | 调皮/俏皮 | 一次性动作 |
| `r13` | 8 | 大笑 | 一次性动作 |
| `r14` | 8 | 哭泣 | 一次性动作 |
| `r15` | 8 | 当前皮肤专属动作 | 默认一次性；若做持久动作，必须另写状态契约与测试 |
| `r16` | 8 | 保留兼容位 | 默认不触发 |
| `r17` | 8 | 保留兼容位 | 默认不触发 |
| `r18` | 8 | 舞蹈 | 菜单触发 |
| `r19` | 8 | 唱歌/舞台 | 菜单触发 |
| `r20` | 8 | 表演/英雄姿态 | 菜单触发 |
| `r21` | 8 | 飞吻/近距离问候 | `c07` 可作为皮肤切换最终姿势 |
| `r22` | 8 | 看手机 | 持久动作，按下退出 |
| `r23` | 8 | 睡眠 | 持久动作，按下或键盘唤醒 |

`r22` 手机状态：

```text
c00 → c01 → c02 → c03        进入
c03 → c04 → c05 → c04 ...   持续循环
c06 → c07                    退出
```

`r23` 睡眠状态：

```text
c00 → c01 → c02 → c03 → c04  入睡并保持
c05 → c06 → c07               唤醒
```

睡眠时显示轻量的 `z` / `Z` 提示，但不得依赖额外位图或文字资源文件。

## 7. 交互状态机

### 7.1 输入优先级

输入按以下顺序处理：

1. 皮肤切换过渡；
2. 睡眠及唤醒；
3. 手机状态及退出；
4. 其他独占持久动作及退出；
5. 普通点击、双击和拖动候选。

低优先级事件不得打断高优先级状态。任何状态结束后必须回到 `r00`，不得停留在过渡帧。

### 7.2 点击与拖动

- 普通单击不播放生气或失败动作；
- 双击播放 `r03` 挥手；
- 指针捕获丢失不触发生气；
- 禁止自动走动或在桌面漫游；
- 拖动阈值为 `max(8px, round(4 × scale))`，225% 时为 9px；
- 达到阈值后播放 `r04 c00`–`c02`；
- 按住期间保持 `r04 c02`；
- 松开后播放 `r11 c02`–`c05`，约 1700ms 后回到待机。

窗口移动使用真实指针增量，拖动期间不得因动画帧尺寸变化而抖动。

### 7.3 凝视

凝视仅在待机状态启用：

- 进入距离：220 个逻辑像素；
- 退出距离：270 个逻辑像素；
- 中心死区：进入 45、退出 34 个逻辑像素；
- 16 个方向，每区 22.5°；
- 方向保持迟滞约 14.75°；
- `r09` 与 `r10` 的帧是方向选择，不是连续动画。

指针离开范围或进入其他动作后立即回到 `r00`。16 个方向帧的身体大小和脚底位置必须一致，只有头部、眼神或局部姿势改变。

### 7.4 持久动作

手机、睡眠和任何皮肤专属持久动作都必须有明确的：

- 进入序列；
- 循环或保持帧；
- 用户退出方式；
- 退出序列；
- 输入抢占规则；
- 回到待机的最终状态；
- 自动测试。

不得只靠定时器猜测动作结束，也不得让普通点击在不同状态中产生不可预测结果。

## 8. 动画平滑与皮肤系统

### 8.1 动作网格

PNG 只保存 176 张作者确认的关键帧。平滑过渡使用双向动作网格文件，例如 17 × 25 的 `.mtn`，逻辑画布为 132 × 202。

每段过渡至少达到：

```text
关键帧数 × ceil(24 / 关键帧数)
```

例如 `r00` 的 6 帧待机循环至少生成 24 个显示阶段。插值使用 smoothstep 或同等平滑曲线。

动画过程中只能有一个不透明主体。禁止把两张完整人物图直接交叉淡入淡出，这会造成双脸、四只手或半透明残影。动作行内共享视口、缩放和脚底注册点。

### 8.2 内置官方皮肤

全部官方皮肤必须嵌入同一个 EXE，并在“皮肤”菜单中即时切换。即使源码旁没有 `skins/` 目录，内置皮肤仍应完整可用。

每个皮肤必须包含：

- 唯一安全标识；
- 176 张 528 × 808 RGBA PNG；
- 完整动作网格；
- 显示名称和资源清单；
- 校验信息。

皮肤切换采用短暂的透明切片式伪 3D 过渡，不使用整张卡片翻转。过渡完成可停在 `r21 c07`，随后回到待机。

### 8.3 外部皮肤兼容

可以支持第三方外部皮肤，但它是兼容能力，不得取代内置官方皮肤。加载前必须：

- 校验标识、路径、ZIP 展开大小和文件数量；
- 拒绝路径穿越和危险扩展名；
- 完整解码所有图片与动作网格；
- 在内存中验证成功后再原子切换；
- 候选皮肤损坏时保留当前皮肤；
- 给出可读错误，不让程序崩溃。

## 9. 实现顺序

为了减少反复修改，按以下阶段一次推进：

### 阶段 A：产品信息

确认角色名称、项目标识、官方皮肤、人物资料、目标仓库和使用边界。建立清单，不生成正式资源。

### 阶段 B：视觉定稿

整理人物资料，制作并确认身份锚点。为 24 个动作行制作联系表，逐行通过人物一致性和动作逻辑检查。

### 阶段 C：工程骨架

创建中性目录 `source/standalone/<character-slug>/`，完成窗口、托盘、菜单、状态机、皮肤接口、自检入口、应用清单和构建脚本。可以参考 `xiaoxiwei` 的实现思路，但不得复制人物专属名称和资源。

### 阶段 D：正式资源

按已批准的联系表生成全部官方皮肤的 176 帧与动作网格。运行尺寸、透明度、命名、数量、解码和人物一致性检查。

### 阶段 E：集成与修复

嵌入资源并运行自动 QA。只修复失败的动作行、皮肤或测试项；除非身份锚点被否决，不重新生成整套资源。

### 阶段 F：发布

在干净的 Windows 环境进行人工验收，生成 EXE、ZIP、SHA-256、发布说明和 QA 证据。首个正式可用版本发布为 `1.0.0`，之后仅在确有功能或修复时递增版本。

## 10. 自动化 QA

程序至少提供：

```text
<character>.exe --self-test
<character>.exe --qa-window
```

`--self-test` 应以非交互方式输出结构化结果，成功时错误列表为空。`--qa-window` 应启动真实窗口、完成基本渲染与退出流程，并适合自动超时检查。

自动化测试至少覆盖：

### 10.1 资源

- 每个官方皮肤恰好 24 行、176 帧；
- 所有 PNG 为 528 × 808 RGBA 且可解码；
- 动作网格存在、格式正确、双向转换可加载；
- `r09`/`r10` 人物大小和脚底锚点一致；
- EXE 内置资源清单与构建清单一致；
- 删除外部皮肤目录后，官方皮肤仍可切换。

### 10.2 状态与交互

- 无自动漫游；
- 单击、双击、拖动阈值、拖动保持和释放恢复正确；
- 16 方向凝视与迟滞正确；
- 手机进入、循环、退出正确；
- 睡眠进入、保持、`z/Z`、唤醒正确；
- 皮肤切换时输入不会破坏状态；
- 所有动作结束后回到待机；
- 暂停、置顶、托盘和 `Ctrl+Q` 正确。

### 10.3 系统行为

- 普通启动单实例；
- 自检和 QA 模式可独立执行；
- 125%、225%、400% 缩放不越界；
- 多显示器和 DPI 变化不丢失窗口；
- 开机启动开/关和重复操作幂等；
- 自动测试不修改真实开机启动设置；
- 损坏外部皮肤不会替换当前皮肤或导致崩溃。

## 11. 人工视觉验收

自动化测试通过后，在干净 Windows 环境人工检查：

- 浅色和深色桌面上的透明边缘；
- 125%、225%、400% 三档缩放；
- 人脸、发型、体型和服装的一致性；
- 全部动作行的脚底稳定、比例稳定和无裁切；
- 无双脸、重复肢体、闪烁、残影或脏边；
- 凝视的上、下、左、右和斜向；
- 拖动、挥手、手机、睡眠与唤醒；
- 每个官方皮肤的菜单切换和动作完整性；
- 托盘、暂停、置顶、开机启动和退出；
- EXE 从不含源码和工具链的目录独立启动。

证据目录至少保存：

- 默认待机；
- 拖动中与释放后；
- 凝视四个主方向；
- 手机持续状态；
- 睡眠与唤醒；
- 每个官方皮肤；
- 三档缩放；
- 自检结果；
- 干净环境启动结果。

## 12. 发布规范

发布包至少包含：

```text
<character>-<version>.exe
<character>-<version>-windows.zip
SHA256SUMS.txt
USER_GUIDE.md
RELEASE_NOTES.md
```

`USER_GUIDE.md` 面向普通用户，只说明：

- 如何启动；
- 如何拖动和触发互动；
- 如何切换官方皮肤；
- 如何设置开机启动；
- 如何退出；
- 常见问题。

不要把外部皮肤开发接口、内部动作网格或构建细节塞入普通用户指南。开发内容留在源码文档。

发布前检查：

- 产品名称、图标、文件名和版本一致；
- 开发者署名为 `Anbunensi`，Company 可留空；
- 不含其他人物名称、旧注册表键或旧资源；
- 不含人物原始照片、视频、音频、歌词、Cookie、Token、密钥或本地绝对路径；
- GitHub Release 有简明介绍、文件清单、使用方式和校验值；

## 14. 失败处理

AI 遇到失败时按范围最小原则处理：

| 失败类型 | 正确处理 |
|---|---|
| 人物不像 | 回到身份锚点，不继续批量生成 |
| 某动作不自然 | 只重做该动作联系表和对应行 |
| 某皮肤资源错误 | 只重做该皮肤对应资源 |
| 编译失败 | 修复工程、清单或编译参数，不重做美术 |
| 状态机失败 | 增加可复现测试后修复对应状态 |
| 干净环境不能运行 | 删除隐式依赖并重新打包 |
| 权利不明确 | 暂停公开发布，保留本地工程并请求确认 |

不要通过连续发布 `v0.x-test1`、`test2`、`final-final` 等版本代替问题定位。未通过 QA 的构建只作为本地构建产物，不创建正式 Release。
