# 女性偶像 Windows 角色生成与开发规范

> 女性角色版，以“小曦薇”v3.0.4 为真实完整样板。本文不是对话记录或项目交接说明，而是一套可重复执行、可逐步自动化的产品开发规范：以后给出一位女性 idol 的名称和目标造型，即可按相同阶段生成一个可独立运行的 Windows 桌面互动角色。

本版动作取自“小曦薇”的实际产品演进，只是一套经过验证的女性角色默认方案，并不规定女性角色必须卖萌、穿裙装或采用某类性格。新项目应优先服从艺人的公开形象、目标角色、服装结构和用户批准；若人物气质更适合利落、酷感、运动或沉静动作，可以直接替换动作脚本，同时保持 24 行、176 个归档帧的技术契约。

> **文档状态**：本规范以已经完成的“小曦薇”v3.0.4 为外部实作案例；本仓库只保存流程文档，不包含“小曦薇”的源码、参考照片、生成帧、皮肤、EXE、QA 制品或私有素材清单。仅克隆本仓库不能重建该成品。

## 1. 规范目标

本规范解决五件事：

1. 还原“小曦薇”从参考素材、写实人物、动作设计、4K 帧、皮肤到单文件 EXE 的真实开发路径。
2. 把一次性的聊天需求整理成稳定的输入、输出、质量门槛和人工确认点。
3. 明确哪些步骤适合自动化，哪些判断必须由粉丝或项目负责人完成。
4. 让不同 idol 项目共享同一套运行时、动作协议、皮肤接口、测试和发布方式。
5. 防止再次出现脸不像、比例跳变、动作重影、帧数过高反而卡顿、拖拽走路不自然等问题。

本文描述的是**目标工作流和当前可执行的半自动 SOP**，不是仓库里已经存在的“一条命令生成任意 idol”功能。`idol-project.yaml`、通用 `idol-character` 命令和数据驱动模板是本规范定义的下一阶段目标；现阶段仍需要私有参考素材、生成工具、人工视觉确认和分阶段构建。

默认产品定位如下：

- Windows 桌面互动角色，而不是必须依附 Codex 运行的宠物。
- 写实或半写实全身人物，透明背景，支持缩放、拖动、待机和换肤。
- 单个 EXE 可独立启动；可选皮肤使用 EXE 同级的 `skins` 文件夹。
- 粉丝个人欣赏、交流、非商业、非官方。
- 不把原始照片、节目视频、歌曲、台词、Cookie、账号凭据打进程序或公开仓库。

## 2. 一条完整流水线

```mermaid
flowchart LR
    A[输入 idol 与目标造型] --> B[参考素材与权利边界]
    B --> C[身份特征与主视觉定稿]
    C --> D[动作关键姿势]
    D --> E[透明化、尺度与脚底注册]
    E --> F[24 行 176 帧契约]
    F --> G[运动网格与 24 阶段补间]
    G --> H[交互状态机与皮肤接口]
    H --> I[自动 QA 与人工动画审核]
    I --> J[单文件 EXE 与完整包]
    J --> K[GitHub 源码与 Release]
```

每个阶段只有在对应门禁通过后才能进入下一阶段。最重要的顺序是：

**先确认脸和全身比例，再批量做动作；先确认动作关键姿势，再生成补间和 EXE。**

如果主脸尚未得到用户批准，禁止一次性生成 176 帧。否则相似度错误会扩散到所有动作和皮肤，返工成本最高。

## 3. 案例研究：“小曦薇”v3.0.4 是怎样开发出来的

### 3.1 从概念到独立产品

最初目标是制作田曦薇的 Codex 桌宠。随后需求逐渐明确为：即使没有 Codex，其他粉丝也能把一个 EXE 发给朋友，双击后直接使用。因此产品从“Codex 宠物图集”演进为 Windows 单文件程序。

这个演进形成了两层资产：

- **原型层**：用 `hatch-pet` 的 Codex v2 图集快速验证人物、基础动作和看鼠标方向。
- **产品层**：从批准的高清动作源条带重新恢复 `528×808` RGBA 关键帧，嵌入 C# WinForms 透明窗口运行时，构建不依赖 Codex、Python 或 OpenCV 的 EXE。

两层不能混用规格。原型图集尺寸较小，适合快速验证；成品必须从高清源重新恢复，不能把小图直接放大，否则会模糊。

### 3.2 主视觉的确认过程

“小曦薇”的主皮肤最终选定为 vivo 活动的白色无袖短裙、黑色细带高跟鞋和长卷发造型。视觉确认经历了以下修正：

1. 第一版偏卡通，无法辨认人物，目标改为写实。
2. 只依赖少量面部参考时脸不像，于是增加正面、三分之四角度、微笑、侧脸等多张同一人物参考。
3. 身材比例异常时，增加正面全身活动截图，按真实头身比重新建立全身尺度。
4. 服装不再凭描述生成，而是使用同一场活动的全身、半身和细节参考统一白裙、鞋、发型和配饰。
5. 用户先批准一张正面全身主视觉，随后所有动作以它作为身份、发色、服装和身体尺度锚点。

这一步的通用结论是：**面部参考负责身份，全身参考负责比例，造型参考负责服装细节；三类素材不可互相替代。**

### 3.3 动作系统的演进

早期方案让人物在鼠标拖拽时随机走路、慢跑或快跑，并在快跑后疲惫、生气。实际预览发现腿部交替不自然，看起来像只有一条腿在动。继续增加帧数不能修复错误的关键姿势，因此最终取消拖拽走跑和自动漫游，改成更符合桌面交互的语义：

- 左键按住并真实移动超过阈值：先吓一跳。
- 拖动期间：维持受惊姿势。
- 松手：皱眉、抬脚、跺脚、抱臂生气。
- 普通轻点、双击挥手、睡眠唤醒、手机退出分别判断，互不误触。

普通待机也从单一眨眼扩展为两级：

- 安静状态：呼吸和眨眼，不持续转头。
- 间隔较长的随机表演：手势舞、唱歌表演、女侠演戏、飞吻、坐下玩手机、侧躺睡觉。

其中“玩手机”和“睡觉”不是播放一次就结束的普通循环，而是持久状态：

- 坐下后持续玩手机，点击才收起手机并起身。
- 侧躺后停留在睡眠姿势，`z/Z` 从头部上方向上飘，点击后播放苏醒动作再起身。

### 3.4 为什么最终使用 24 个显示阶段

最早每个动作只有约 6 个关键姿势，动作跨度大。随后尝试把循环扩展到 56 个显示阶段，虽然理论上更细，但实际出现卡顿、人物残影和视觉拖沓。最终方案是：

- 只保存 4–8 个经过人工审核的高清关键姿势。
- 构建时为相邻关键姿势生成双向稠密运动场，再压缩为 `17×25` 运动网格。
- 运行时用 smoothstep 缓动，把完整动作呈现为至少 24 个显示阶段。
- 保持 v3.0.1 已确认的动作总时长，不因为阶段数下降而整体加速。
- 每个中间阶段只选择一个不透明人物作为所有者并做形变，不把两张整图做 alpha 交叉淡化。

当前映射是：8 张关键姿势每段 3 阶，完整循环 24 阶；6 张每段 4 阶，完整循环 24 阶；5 张每段 5 阶，完整循环 25 阶；4 张每段 6 阶，完整循环 24 阶。关键姿势不是越多越好，数量取决于动作中真正不可由形变推导的姿态变化。

最后一条是消除“双人重影”的关键。透明交叉淡化会同时显示前后两个完整人物，尤其是腿、手和头移动较大时非常明显；单一姿态形变则始终只有一个轮廓。

### 3.5 皮肤与换装

在内置 vivo 白裙之外，项目增加了两套完整外置皮肤：

- 临安公主｜《大奉打更人》，专属动作“公主拂袖行礼”。
- 黄橙子｜《如此可爱的我们》，专属动作“橙子放学元气招呼”。

所有皮肤共享相同动作协议，因此换皮肤不改变运行时代码。每套外置皮肤仍包含完整 176 张关键帧，`skin.xml` 的 `exclusiveAction` 决定第 15 行是否在菜单和随机待机中启用。

最初换肤像整张卡片翻转，不像人物变装。v3.0.4 改为只处理人物 alpha 轮廓：将身体纵向切成 16 个水平切片，以宽度压缩、水平偏移和明暗变化模拟人物转身；旋转半圈中由头到脚换成新皮肤，最后定格 pose，再平滑回到待机。整个透明窗口从不翻转。

### 3.6 最终产品化

成品统一命名为“小曦薇”，而不是“小曦薇桌宠”：

- EXE：`小曦薇.exe`
- 完整包：`小曦薇-完整包.zip`
- 开发者：`Anbunengsi`
- Company：留空
- 当前版本：`3.0.4`
- GitHub 仓库：`super-idol-hub/xiaoxiwei`

源码仓库只保存运行时代码、构建脚本、接口和说明。用户照片、网络参考图、生成中间图、动作条带、QA 缓存和大型二进制不提交 Git；EXE 和完整包通过 GitHub Releases 发布。

## 4. 新 idol 项目的输入协议

以后不应只接收一句模糊的“做某某桌宠”就立刻生成全部素材。自动化入口可以从一句自然语言开始，但系统应整理为以下项目清单。

### 4.1 最小自然语言请求

推荐格式：

> 为「偶像名」生成一个 Windows 桌面互动角色。主造型是「活动/剧集/服装」，整体「写实/半写实」；附加皮肤是「角色或其他造型」；开发者显示为「名字」；仅供粉丝非商业使用。

若用户只说“为某 idol 生成”，系统可以先搜索公开可见的活动或角色造型候选，但在主造型批准前不得批量生成动作。

### 4.3 必需参考素材

一个可靠的主皮肤建议至少准备：

- 正面或接近正面的清晰面部 2–4 张。
- 左右三分之四角度各 1–2 张。
- 自然微笑、闭眼笑、严肃或生气等表情 2–4 张。
- 正面全身 1–2 张，必须能看到腿和鞋。
- 同套造型半身或细节图 2–4 张。
- 发饰、耳饰、鞋、裙摆等容易漂移的局部参考。

优先使用同一时期、同一妆发的素材。跨年份或差异很大的妆造会让生成模型把五官和发型平均化，降低辨识度。

每张参考应在私有来源清单中记录：文件名、来源 URL、获得日期、用途、是否可公开。公开仓库只保留文字说明，不保留原始图。

## 5. 标准阶段、产物和门禁

### 阶段 0：产品边界与命名

**输入**：idol 名称、产品显示名、ASCII slug、开发者、使用范围、是否公开。

**操作**：

1. 产品显示名可以中文；代码命名、目录、仓库和皮肤 ID 使用 ASCII slug。
2. 先确定是个人自用还是公开发布。公开发布必须有免责声明和下架渠道。
3. 默认不使用“官方”“授权”“代言”等可能造成误解的表述。
4. 默认不启用商业用途，也不自动选择允许商业使用的开源许可证。

**输出**：项目清单、文件名、仓库名、初始版本号、免责声明草案。

**人工门禁 A**：用户确认显示名、造型方向、开发者和公开范围。

### 阶段 1：研究和素材登记

**输入**：idol 名称、目标活动/剧集/角色。

**操作**：

1. 搜索公开可见的正脸、三分之四角度、全身和服装细节。
2. 优先原活动、工作室、品牌公开页或清晰媒体图；粉丝修图只能辅助判断，不能作为唯一身份基准。
3. 不下载或打包节目视频、音乐、台词、字幕和水印素材。
4. 将参考放入私有 `user-materials` 或专用私有素材目录，写入来源清单并加入 `.gitignore`。
5. 从素材中提炼可描述的身份特征，而不是复制某一张照片的背景和构图。

**输出**：参考集合、素材来源登记、目标造型说明、风险项。

**自动检查**：私有路径已被 Git 忽略；公开目录中没有原图、Cookie、Token 或绝对用户路径。

**人工门禁 B**：用户选定唯一主造型，并确认哪些附加造型以后作为皮肤。

### 阶段 2：建立身份与尺度基准

**输入**：批准的参考集合。

**操作**：先生成或重建一张正面站立全身主视觉。需要锁定：

- 脸型、眼距、眼型、鼻唇关系、笑容和下颌线。
- 头发颜色、刘海分区、卷度、长度和发饰位置。
- 真实头身比、肩宽、腰线、腿长和脚部大小。
- 服装领口、腰线、裙摆层级、鞋型和配饰。
- 正面自然站姿、双脚完整、手指不过度扭曲。

成品人物必须放在纯色或可干净抠像的背景上，并保留足够透明边距。不要在这一步同时生成十几种动作。

**输出**：`identity-anchor`、`neutral-fullbody`、身份特征表、尺度/脚底基线。

**自动检查**：图像清晰度、完整全身、边缘未裁切、背景可分离。

**人工门禁 C（最重要）**：用户明确回复主脸、身材比例、衣服、鞋和发型均可接受。未通过只修主视觉。

### 阶段 3：动作设计表

**输入**：批准的主视觉、idol 的公众形象和用户偏好。

每个动作先写成动作脚本，再生成图片。动作脚本至少包括：

- 动作名称和使用场景。
- 开始、中间、结束关键姿势。
- 哪只手、哪只脚先动。
- 重心、视线和表情变化。
- 是否循环、循环区间、退出方式。
- 建议关键帧数和动作总时长。

例如“拖拽后生气”不是简单写“生气”，而是：皱眉准备 → 抬脚 → 跺脚落地 → 抱臂鼓嘴 → 回到待机。这样才能避免动作只靠一条腿重复。

动作设计遵循三条原则：

1. 先让关键姿势本身成立，再考虑补间。
2. 大幅改变腿部、手臂或身体朝向时，必须增加真正不同的关键姿势，不能用插值伪造完整步态。
3. 如果一个动作在小尺寸桌面上难以辨认，改成更清晰的交互反应，不要执着于动作名称。

**输出**：动作表、行号计划、关键姿势提示词、持续状态定义。

**人工门禁 D**：确认动作符合 idol 气质，不冒犯、不恶搞，并确认是否需要删除走路/跑步等高风险动作。

### 阶段 4：关键帧与透明化

**输入**：身份锚点、动作表。

**操作**：

1. 每个动作通常生成 4–8 张真正不同的关键姿势。
2. 每次生成都引用同一身份锚点、服装说明和全身比例说明。
3. 批次生成后逐张比较脸、发色、鞋、裙摆和身体尺度。
4. 清除背景、色键残留、半透明脏边和透明像素中的非零 RGB。
5. 把每一帧注册到统一画布、脚底线和人体中心，不按单帧人物边框自动缩放。
6. 同一动作相邻帧只允许姿势变化，不允许人物整体忽大忽小。

小曦薇的高清恢复不是把原型图集放大。原型每格为 `192×208`，恢复时回到原始动作条带，以 4× 几何计算 `768×832` 全格，再使用统一的 `(120, 12, 648, 820)` 裁剪框得到 `528×808`。同一动作整行共用 viewport、缩放和脚底线，使用 LANCZOS 重采样；禁止逐帧执行“自动铺满画布”。

内置白裙素材曾使用青色键 `(0,255,255)` 和确定性 despill 清理。该颜色不是通用默认值：如果新皮肤本身包含青色或浅蓝服装，应改用与服装不冲突的背景键色，并把阈值、去溢色参数和例外写入项目清单，不能复制人物专用的魔法颜色或坐标。

**输出**：高清 RGBA 动作条带或独立关键帧、动作联系表、透明化报告。

**自动检查**：尺寸一致、RGBA、非空、无边缘 alpha、无透明 RGB 残留、脚底/人体中心偏差在阈值内。

**人工门禁 E**：逐行查看联系表，重点检查脸、发色、手指、鞋、腿、裙摆、脚底、尺度跳变和背景残留。

### 阶段 5：视线方向

看鼠标不是把整个人缩放或旋转，而是 16 个方向的眼神变化加非常轻微的头部转向。

方向语义使用固定角度：

- `000°`：上。
- `090°`：右。
- `180°`：下。
- `270°`：左。
- 其余方向以 `22.5°` 等分补齐。

制作时先批准上、右、下、左四个基准方向，再补齐 16 个方向。所有方向必须与普通待机保持同一人物高度、脚底线和下半身中心。

运行时采用进入/退出迟滞：小曦薇当前是在约 `220` 逻辑像素范围内进入跟随，移动到约 `270` 逻辑像素外才退出；中心还有死区，方向扇区也有保持角，防止鼠标轻微移动导致头部抖动。退出后立即恢复普通呼吸眨眼。

**人工门禁 F**：进行盲测，不显示方向标签，让审核者判断人物究竟在看哪个方向；比例和脚底也必须稳定。

### 阶段 6：成品 24 行、176 帧协议

“小曦薇”独立版使用固定 `528×808` RGBA 关键帧。目录为 `rNN/cNN.png`，24 行共 176 张：

| 行 | 帧数 | 语义 | 备注 |
|---:|---:|---|---|
| 0 | 7 个归档槽位 | 普通待机/呼吸/眨眼 | 运行时循环 6 个关键帧；`c06` 是 `c00` 的兼容静止副本 |
| 1 | 8 | 旧版向右移动 | v3.0.4 保留兼容，运行时不启用 |
| 2 | 8 | 旧版向左移动 | v3.0.4 保留兼容，运行时不启用 |
| 3 | 4 | 挥手 | 双击或菜单触发 |
| 4 | 5 | 跳起/受惊 | 拖拽超过阈值后使用前半段 |
| 5 | 8 | 疲惫/委屈 | 手动动作 |
| 6 | 6 | 等待回应 | 手动动作 |
| 7 | 6 | 认真工作 | 手动动作 |
| 8 | 6 | 检查/回应 | 手动动作 |
| 9 | 8 | 看鼠标方向 0–7 | 直接方向，不做普通动作循环 |
| 10 | 8 | 看鼠标方向 8–15 | 与第 9 行合计 16 方向 |
| 11 | 6 | 生气跺脚 | 拖拽松手后从皱眉段开始 |
| 12 | 8 | 内置皮肤卖萌 | 仅内置主皮肤启用 |
| 13 | 8 | 内置皮肤大笑 | 仅内置主皮肤启用 |
| 14 | 8 | 内置皮肤哭 | 仅内置主皮肤启用 |
| 15 | 8 | 外置皮肤专属动作 | 名称由 `exclusiveAction` 声明 |
| 16 | 8 | 保留 | 兼容位，不应随意改语义 |
| 17 | 8 | 保留 | 兼容位，不应随意改语义 |
| 18 | 8 | 手势舞 | 公共随机待机 |
| 19 | 8 | 唱歌/舞蹈表演 | 不内置音乐或歌词 |
| 20 | 8 | 演戏/女侠动作 | 公共随机待机 |
| 21 | 8 | 飞吻/直播 pose | 换肤结束 pose 也取该行末帧 |
| 22 | 8 | 坐下玩手机 | 0–3 进入，3–5 循环，6–7 退出 |
| 23 | 8 | 侧躺睡眠 | 0–4 入睡，4 保持，5–7 苏醒 |

所有外置皮肤都必须提供完整 176 个归档帧，包括当前保留行；其中第 0 行实际只创作 6 个运行时关键姿势，`c06` 复制 `c00`。这能保证皮肤包与运行时版本独立，避免切换后某个动作突然缺帧。

### 阶段 7：运动网格与 24 阶段补间

高清关键帧先缩到 `132×202` 逻辑空间计算双向 Farneback 光流，再采样为 `17×25` 网格，按 64 倍量化为 int16 并写入 `XWM1` 格式的 `.mtn`。小曦薇使用的光流参数为 `pyr_scale=0.5`、`levels=5`、`winsize=25`、`iterations=5`、`poly_n=7`、`poly_sigma=1.5`，位移钳制在逻辑空间 ±48 像素。运行时按窗口大小扩展网格并形变高清帧。

补间规则：

- 完整动作至少 24 个显示阶段，不要求保存 24 张 PNG。
- 使用 smoothstep 缓动，减轻关键姿势处的速度突变。
- 保持已经确认的动作总时长。
- 看鼠标的第 9、10 行是直接方向，不生成普通循环补间。
- 中间帧只渲染一个姿态所有者，`crossFade = false`。
- 皮肤 API 的最低兼容要求允许运动网格缺失，运行时会退回单一端点姿态；但官方随发行包提供的内置和外置皮肤必须生成完整运动网格并通过 QA。

当前 176 张关键姿势会生成 313 组运动网格，包括各普通动作的相邻循环和非待机动作回到待机的过渡；第 9、10 行视线方向不参与这一循环补间。

睡眠的 `z/Z` 粒子仍可使用独立的 56 阶段效果，因为它只绘制轻量文字粒子，不等于让整个人物动作使用 56 帧。换肤过渡使用独立的 60 阶段、约 1.2 秒时间线，也不计入普通动作的 24 阶段预算。

### 阶段 8：交互状态机

建议把行为实现为明确状态，不要用相互覆盖的计时器拼凑：

- `Idle`：普通呼吸眨眼。
- `MouseLook`：仅近距离启用，离开范围回 `Idle`。
- `Waving`、`Waiting`、`Working` 等：一次性临时动作。
- `DragStartled`：真实拖拽时的受惊进入与保持。
- `AngryStomp`：拖拽松手后的生气动作。
- `SittingPhone`：进入、持久循环、点击退出三段。
- `SideRest`：入睡、持久睡眠、点击苏醒三段。
- `SkinTransition`：锁定动作输入，候选皮肤成功预热后原子切换。
- `Paused`：停止动画但保留安全退出和菜单。

运行时处理优先级应固定为：暂停 → 换肤 → 拖拽反应 → 持续睡眠 → 持续坐姿 → 鼠标注视 → 长关键帧停留 → 普通动作补间。高优先级状态负责消费输入并阻止低优先级动作误触发。

小曦薇的随机待机在普通空闲约 12–25 秒后触发；特殊待机播放完后，下一次间隔约 18–45 秒。新项目可以配置这些区间，但普通眨眼应始终占据绝大多数安静时间。

拖拽阈值必须把普通点击和鼠标按下抖动过滤掉。小曦薇当前最小阈值为 8 像素，并按显示缩放适当调整。

### 阶段 9：皮肤接口

外置皮肤目录：

```text
小曦薇.exe
skins/
  <skin-id>/
    skin.xml
    frames.zip
```

`skin.xml` 示例：

```xml
<?xml version="1.0" encoding="utf-8"?>
<skin apiVersion="1"
      id="linan-princess"
      name="临安公主｜大奉打更人"
      developer="Anbunengsi"
      archive="frames.zip"
      exclusiveAction="公主拂袖行礼" />
```

接口约束：

- 文件夹名必须与 `id` 完全一致，ID 只允许 `[A-Za-z0-9._-]`。
- `apiVersion` 当前固定为 `1`，`archive` 固定为 `frames.zip`。
- 文本字段最长 64 字符。
- ZIP 根目录直接包含 `r00/c00.png` 等帧；可选运动网格位于 `motion/...mtn`，不能再套一层 `frames/`。
- 每张 PNG 必须为 `528×808` RGBA。
- 单条目最大 32 MiB；压缩包和总解压大小各最大 128 MiB。
- 禁止重复条目、绝对路径、`..`、反斜杠路径和未知文件。
- 无效皮肤静默跳过，程序回退到内置皮肤。
- 候选皮肤先独立解码、验证和预热；换肤完成时原子交换，不能让半加载皮肤进入当前状态。

专属动作只放在第 15 行。公共动作仍应全部存在，保证换肤后交互逻辑不变。

这里需要区分两档要求：第三方皮肤兼容底线是 176 张合法 PNG，`.mtn` 可以没有；由项目官方打进完整包的皮肤则要求 313 组 `.mtn` 全部生成、可解析并通过单轮廓 QA。

### 阶段 10：可选的公开动态提醒

动态提醒不是角色生成的必需部分，应作为可关闭模块。小曦薇目前读取新浪公开媒体页：

- 不登录、不使用 Cookie、不抓取私人内容。
- 启动后首次检查只建立最新 ID 基线，不把历史内容当成新动态。
- 正常每 9–11 分钟随机检查一次。
- 请求超时 12 秒，响应最大 2 MiB。
- 失败时静默指数退避，最长约 120 分钟，不弹出断网错误。
- 只保存提醒开关和最后看到的 ID。
- 仅允许受信任的 HTTPS 主机和固定动态链接格式。

为其他 idol 生成时，必须把账号页面、链接解析规则和受信任主机做成配置，而不是复制田曦薇的硬编码地址。平台页面结构可能随时变化，因此提醒只能标注为“尽力而为”，不能承诺实时。

### 阶段 11：Windows 运行时

当前运行时是一个 C# WinForms 单文件源程序，核心能力包括：

- 无边框、逐像素透明、始终置顶的 layered window。
- 托盘图标、右键菜单、暂停、重置位置和退出。
- 单实例互斥锁，避免重复启动多个角色。
- 左键拖动、双击挥手、滚轮缩放、`Ctrl+Q` 退出。
- 受限 LRU 帧缓存；当前预算 128 MiB。
- 内置帧 ZIP 作为程序集资源，外置皮肤从 EXE 同级读取。
- 自检模式 `--self-test` 和限时窗口 QA 模式 `--qa-window`。
- 崩溃日志写入临时目录，不在桌面制造额外配置文件。

默认显示比例是 225%，菜单提供 125%、175%、225%、300% 和 400%。400% 对应 `528×808` 源帧的原始显示尺度。

单文件表示不需要旁置业务 DLL、Codex、Python 或 OpenCV；程序仍使用 Windows 自带或系统安装的 .NET Framework 4.x 组件。以后若迁移到现代 .NET 自包含发布，应重新评估 EXE 体积、Windows 版本和逐像素透明窗口兼容性。

### 阶段 12：构建

完整本地工作区中的主要脚本：

| 文件 | 职责 |
|---|---|
| `source/standalone/xiaoxiwei/XiaoXiWeiPet.cs` | 窗口、状态机、渲染、皮肤、提醒、自检 |
| `build_hd_frames.py` | 从批准的高清动作条带恢复、注册并验证 176 张帧 |
| `build_motion_fields.py` | 生成双向运动网格和无重影 QA |
| `build_skin.ps1` | 重建、验证并打包一套外置皮肤 |
| `pack_external_skin.py` | 强制执行皮肤 ZIP 和元数据安全协议 |
| `render_runtime_smoothing_qa.py` | 离线模拟运行时补间，输出 GIF、联系表和报告 |
| `build.ps1` | 生成帧、运动网格并调用 C# 编译器生成 EXE |
| `package_project.ps1` | 汇总最新版、皮肤、文档、哈希、历史版本和完整包 |
| `app.manifest` | Windows 清单 |
| `xiaoxiwei.ico` | EXE 和托盘图标 |

主程序构建顺序：

1. `build_hd_frames.py` 生成 `frames-4k.zip` 和帧 QA。
2. `build_motion_fields.py` 生成 `.mtn` 并追加到资源 ZIP。
3. `csc.exe` 编译 `XiaoXiWeiPet.cs`，嵌入帧 ZIP、图标和 manifest。
4. 输出单文件 EXE。

当前 `build.ps1` 只完成生成与编译，并没有自动串联 EXE 自检和窗口烟测。标准化后的发布命令必须在编译后显式执行 `--self-test`、`--qa-window`、外置皮肤校验和从干净目录解压启动，任一失败都不得打包 Release。

Python、Pillow、NumPy 和 OpenCV 只用于构建。OpenCV 生成运动网格后不会成为 EXE 运行时依赖。

下面的命令只适用于“小曦薇”原始私有工作区布局，不适用于刚克隆、缺少素材的公开仓库：

```powershell
cd <完整私有工作区>\standalone\xiaoxiwei
.\build.ps1
```

原始脚本按自身位置向上定位工作区，因此要求同一根目录下至少存在：

```text
<完整私有工作区>/
  standalone/xiaoxiwei/             # C# 与构建脚本
  work/xiaoxiwei/realistic-run/     # final 低清基准与 decoded 动作条带
  work/xiaoxiwei/standalone-4k/     # 生成的高清帧和 motion
  outputs/xiaoxiwei-standalone-4k-v3/
```

公开仓库中的脚本副本位于 `source/standalone/xiaoxiwei`，而私有 `work` 与大型帧包被正确忽略；若直接运行会因相对根目录和输入缺失而失败。未来模板必须让工作区根目录和私有素材目录成为显式参数，不能继续依赖脚本复制位置。

生成皮肤示例：

```powershell

  -SkinId 'linan-princess' `
  -SkinName '临安公主｜大奉打更人' `
  -DecodedDir '<该皮肤高清动作条带目录>' `

```

注意：公开 GitHub 仓库正确地排除了参考照片、`work`、生成帧包和大型制品，因此**仅克隆公开仓库不能完整复现同一人物 EXE**。公开源码可以复现运行逻辑和构建结构；完整重建仍需要私有、已批准的动作源输入以及本地 `hatch-pet` 辅助工具。未来若要做到“一键生成”，应把通用辅助函数提取为版本化构建工具，同时继续把具体人物参考和生成帧留在私有制品库。

### 阶段 13：自动 QA

每次构建必须生成机器可读报告，并至少检查：

**帧协议**

- 24 行、176 张已用帧均存在。
- 每张为 `528×808` RGBA 且非空。
- 没有画布边缘裁切、透明 RGB 污染或异常色键残留。
- 人物高度、下半身中心、脚底线和头部区域在允许范围内。
- 看鼠标 16 方向与待机比例一致。

**运动**

- 对官方发行的内置和外置皮肤，313 组相邻/回待机运动网格完整；第三方最低兼容包可缺少，但必须明确报告已进入回退路径。
- 中间姿势仍有不透明核心，最大人物连通区域占比不低于阈值。
- 中间帧只有一个人物轮廓，没有整图交叉淡化。
- 每个动作达到至少 24 个显示阶段并保留总时长。

**皮肤**

- `skin.xml` 字段、ID、文件夹和 ZIP 名称正确。
- 无路径穿越、重复条目、未知文件和解压炸弹风险。
- 176 个归档帧必须齐全；官方发行皮肤还必须包含完整运动网格，第三方兼容包缺少网格时明确记录回退。
- 第 15 行和 `exclusiveAction` 一致。
- 损坏皮肤不会阻止内置皮肤启动。

**运行时**

- 自检成功并能解码 176 帧。
- 透明窗口能打开并在指定时间自动退出。
- 单实例、缩放、拖拽、双击、暂停、托盘和退出有效。
- 睡眠、手机和换肤状态能正常结束。
- 网络断开时提醒模块静默失败。

自检命令：

```powershell
.\小曦薇.exe --self-test .\self-test.json .\self-test-preview.png
.\小曦薇.exe --qa-window 8000
```

当前 `render_runtime_smoothing_qa.py` 仍保留旧版 opacity crossfade 的部分描述和模拟方式，与 v3.0.4 的“单一轮廓 + motion mesh”运行时并不完全一致。因此现阶段权威结论应来自 `motion-build-report.json`、EXE `--self-test` 和真实 `--qa-window`；该离线脚本在重写为同一渲染算法前只能作为辅助预览，不能独立决定发布是否通过。

### 阶段 14：人工视觉 QA

自动指标不能判断“像不像”或“动作是否像本人”。必须查看以下制品：

- 每行关键姿势联系表。
- 每个动作的透明背景 GIF。
- 深色和浅色背景预览。
- 看鼠标 16 方向盲测图。
- 运动网格的前帧/中间帧/后帧对照。
- 皮肤切换完整录像或逐阶段联系表。
- 125%、225% 和 400% 三种大小的实际桌面截图。

人工审核问题：

1. 第一眼能否辨认 idol？
2. 所有动作是否保持同一张脸、同一发色、同一妆发？
3. 人物是否在某一帧突然变高、变矮、变宽或脚底跳动？
4. 手、脚、裙摆和头发是否在补间中断裂或出现第二个轮廓？
5. 动作是否清楚、自然，并符合人物气质？
6. 普通待机是否安静，随机表演是否不过于频繁？
7. 鼠标移远后是否停止盯视并恢复眨眼？
8. 换肤是否像人物变装，而不是整张窗口翻卡片？

**人工门禁 G**：动画、交互和皮肤全部批准后才允许生成公开 Release。

### 阶段 15：打包与发布

本地发布结构建议：

```text
<项目>/
  README.md
  CHANGELOG.md
  RELEASE_NOTES.md
  PROJECT-MANIFEST.json
  docs/
  source/
  skin-pack-template/
  releases/
    最新版/
      <显示名>.exe
      skins/
      使用说明.md
      皮肤包接口.md
      SHA256SUMS.txt
    历史版本/
    <显示名>-完整包.zip
  qa/
```

公开 Git 仓库保存源码和文档；以下内容默认忽略：

- `releases/` 和 `qa/`。
- 用户原图、网络参考、生成中间素材和私有来源清单。
- `work/`、内置帧 ZIP、EXE、DLL、PDB、缓存和日志。

GitHub 发布步骤：

1. 确认版本、程序集元数据、README、CHANGELOG 和 Release Notes 一致。
2. 在干净 Windows 用户环境烟测 EXE 和完整包。
3. 生成 EXE、皮肤和完整包的 SHA-256。
4. 扫描 Git 暂存区，确保无原始参考、绝对路径、凭据和大型二进制。
5. 提交源码和文档；创建版本 tag。
6. 创建 GitHub Release，上传完整 ZIP、必要的独立 EXE 和校验值。
7. 从 Release 重新下载一次，在新目录解压复测。

公开仓库、创建 tag 和上传 Release 都是外部写入，自动化工具必须在执行前取得项目负责人明确授权。

## 6. 自动化边界

### 6.1 可以自动化

- 把自然语言需求整理成 `idol-project.yaml`。
- 生成标准目录、README、免责声明和版本文件骨架。
- 搜索公开参考候选并生成私有来源清单。
- 背景移除、色键清理、透明边缘清理。
- 统一画布、脚底线、人物中心和尺度注册。
- 生成动作联系表、GIF、对照图和方向盲测材料。
- 验证 176 帧、尺寸、RGBA、ZIP 路径和安全上限。
- 生成双向运动网格和 24 阶段运行时补间。
- 构建 EXE、皮肤包、完整 ZIP、manifest 和 SHA-256。
- 执行自检、QA 窗口、隐私扫描和 Git 大文件扫描。
- 在获得明确授权后创建提交、tag 和 Release 草稿。

### 6.2 必须人工确认

- 主脸是否像、全身比例是否正确。
- 服装、鞋、发型、配饰和时代/角色造型是否正确。
- 动作是否符合 idol 气质，是否可能造成冒犯或误解。
- 手势舞、唱歌或演戏动作是否过度复刻受保护的具体表演。
- 走路、跑步等复杂动作是否真的自然；不自然时应删除而不是强行保留。
- 每套皮肤的专属动作和换肤效果。
- 公开参考和生成素材的权利风险。
- 免责声明、许可策略、仓库公开/私有和最终发布。
- 任何权利方提出的下架或修改请求。

### 6.3 推荐的人机门禁

| 门禁 | 人工批准内容 | 未通过时的修复范围 |
|---|---|---|
| A | 产品名、主造型、开发者、公开范围 | 只改项目清单 |
| B | 参考集合和皮肤候选 | 只补/换参考 |
| C | 主脸、体型、衣服、鞋、发型 | 只重建主视觉，不做动作 |
| D | 动作脚本和人物气质 | 只改动作设计 |
| E | 每行动作联系表 | 只修失败行或失败帧 |
| F | 16 方向视线与比例 | 只修第 9、10 行 |
| G | EXE、皮肤、免责声明和发布包 | 只修对应模块，复测后发布 |

## 7. 从“小曦薇”得到的故障库

| 现象 | 根因 | 标准修复 | 不推荐做法 |
|---|---|---|---|
| 太卡通、认不出 | 风格和身份参考不足 | 切换写实目标，补正面与三分之四面部参考 | 只在提示词中反复写“更像” |
| 脸仍不像 | 单张或跨时期素材造成平均脸 | 建身份锚点，锁定五官关系后再批量动作 | 先生成全部动作再逐帧修脸 |
| 身材比例怪 | 缺少正面全身和脚部参考 | 用同套全身图重建头身比、腿长和鞋 | 从半身图猜完整身体 |
| 成品模糊 | 从小尺寸图集直接放大 | 从批准的高清源条带恢复 4× 帧 | 对 192×208 小格做简单插值放大 |
| 动作跳得很大 | 关键姿势少且跨度过大 | 增加真正必要的关键姿势，再做 24 阶段网格补间 | 只复制相同帧凑数量 |
| 56 阶段卡顿 | 人物全帧和缓存压力过高 | 普通循环收敛到至少 24 阶段并保留总时长 | 认为阶段越多必然越流畅 |
| 出现双人重影 | 前后整帧透明交叉淡化 | 单一不透明姿态所有权 + 运动网格形变 | 同时绘制两个人物轮廓 |
| 走跑像一条腿动 | 关键步态和重心错误 | 若无法重做完整步态，取消移动，改受惊→生气 | 用更多补间掩盖错误姿势 |
| 手势舞时发色变化 | 批次身份/色彩漂移 | 锁定身份锚点和发色，单独修该动作行 | 重建所有皮肤和运行时 |
| 看鼠标时人物缩小 | 方向帧用了单帧自动缩放或不同注册 | 第 9、10 行复用待机尺度、脚底和下半身中心 | 每个方向按自身边框铺满画布 |
| 一直盯着鼠标 | 没有距离边界和退出迟滞 | 设进入/退出半径、死区和离开后普通待机 | 让全屏鼠标永久驱动头部 |
| 待机一直转头 | 普通待机和鼠标跟随混在一起 | 普通状态只呼吸眨眼，转头仅在近距跟随 | 把转头眨眼当作默认循环 |
| 睡眠 `z/Z` 位置不对 | 粒子锚定窗口而非人物头部 | 从当前可见人物边界计算头部发射点 | 使用固定屏幕坐标 |
| 玩手机/睡觉自动起身 | 没有持久状态和独立退出段 | 定义进入、保持循环、点击退出 | 把整行当普通一次性动画 |
| 换肤像卡片翻转 | 旋转整张透明画布 | 只切片人物 alpha 轮廓，伪 3D 转身并由头到脚替换 | 对窗口做矩形 3D 翻牌 |
| 皮肤加载一半失败 | 直接替换当前缓存 | 候选皮肤完整验证和预热后原子交换 | 边读边替换活动皮肤 |

修复原则是“最小失败范围重建”：发色问题只修对应动作行，视线缩放只修第 9、10 行，某套皮肤只修该皮肤。只有身份锚点本身错误时才重新生成全套。

## 8. 两套技术规格不要混淆

### 8.1 Codex v2 原型图集

`hatch-pet` 的标准 Codex v2 宠物使用：

- 8 列 × 11 行图集。
- 每格 `192×208`。
- 最终图集 `1536×2288`。
- `spriteVersionNumber: 2`。
- 第 0–8 行为基础动作，第 9–10 行为 16 个看鼠标方向。

它适合快速在 Codex 中验证人物和基础交互，也可作为小曦薇高清恢复脚本的低分辨率几何参考。

### 8.2 4K 独立 Windows 产品

小曦薇独立版使用：

- 24 行、每行最多 8 列。
- 共 176 张已用关键帧。
- 每张 `528×808` RGBA。
- 运行逻辑尺寸 `132×202`。
- 双向 `17×25` 运动网格。
- 普通动作至少 24 个运行时显示阶段。
- 外置皮肤 API v1。

以后若目标是“所有粉丝都能双击使用”，最终交付应以第二套规格为准。不能把第一套小图集直接重命名或放大后当作 4K 成品。

## 9. 项目模板和目录建议

建议把通用生成器与具体 idol 项目分开：

```text
idol-windows-character-toolkit/       # 可公开的通用生成器
  runtime/
  builders/
  schemas/
  templates/
  qa/

projects/<idol-slug>/                 # 具体项目
  idol-project.yaml
  public/                             # 可提交源码和文档
  private/user-materials/             # 参考原图与来源清单，不提交
  work/                               # 生成条带、注册帧和运动网格，不提交
  outputs/                            # EXE、皮肤和 QA 制品，不提交
  release/                            # 最终分发包
```

通用运行时不应包含 `XiaoXiWei`、固定微博 URL、固定免责声明或固定皮肤名。自动生成新项目时替换：

- 命名空间、互斥锁、错误日志目录和本地设置目录。
- AssemblyTitle、Product、Description、Version、Developer。
- 窗口标题、菜单、托盘提示、使用说明和免责声明。
- 内置帧资源名、图标、默认比例和产品文件名。
- 社交动态数据源和受信任主机。
- 主皮肤和专属动作菜单名。

## 10. 完成定义

只有同时满足下列条件，才算一个新 idol Windows 角色已经完成：

- 用户明确批准主脸、体型、服装、鞋和发型。
- 24 行 176 张关键帧协议完整，所有皮肤同协议。
- 所有动作在联系表和实际桌面尺寸下都自然且能辨认。
- 看鼠标只在近距工作，离开后恢复呼吸眨眼。
- 拖拽、双击、手机、睡眠和换肤手势互不冲突。
- 普通动作至少 24 个显示阶段，无双人重影和明显卡顿。
- 手机和睡眠为持久状态，点击后走完整退出动作。
- `z/Z` 从人物头部上方飘出。
- 换肤只转人物轮廓，成功验证后原子切换。
- 皮肤损坏或网络断开不会导致主程序无法使用。
- EXE 在没有 Codex、Python 和 OpenCV 的 Windows 环境可独立启动。
- 自检、窗口 QA、哈希、隐私扫描和干净环境烟测通过。
- README、版本、免责声明、Release Notes 和程序集元数据一致。
- 公开仓库不含参考原图、私有路径、凭据或大型生成制品。
- 对外分发包有明确非官方、非商业声明和权利方处理渠道。

## 11. 当前样板的已知后续改进

小曦薇 v3.0.4 已可作为可用样板，但要把流程升级为真正的“一句话生成任意 idol”，还应完成：

1. 把 C# 中的产品名、文本、动作名、社交地址、版本和免责声明全部移入项目清单或代码生成模板。
2. 把 `build_hd_frames.py` 中与小曦薇具体动作条带名称绑定的部分改成数据驱动映射。
3. 把本地 `hatch-pet` 辅助函数封装成版本化工具，消除公开构建脚本对个人环境路径的依赖。
4. 为 `idol-project.yaml` 编写 schema 校验器和项目脚手架命令。
5. 增加自动隐私扫描、帧视觉异常检测、动画性能基准和干净虚拟机烟测。
6. 把账号监控抽象成可关闭的数据源插件，不在通用模板中硬编码某位 idol。
7. 明确源码许可与人物肖像/生成视觉资产的权利边界。当前项目未提供通用开源许可证；在坚持禁止商用的前提下，不应直接套用允许商业使用的 MIT、Apache 或 GPL。
8. 增加清晰的权利方联系渠道，例如 GitHub Issues 或专用邮箱。
9. 将 24 行语义、帧数、动作时长、循环区间、尺寸和 ZIP 限额收敛为唯一的 `action-contract.json`；目前同一协议在 C#、多个 Python 脚本和文档中重复，容易版本漂移。
10. 将约 5700 行的单体 C# 源文件拆成通用运行时模块，例如窗口渲染、动画控制、交互控制、皮肤目录、动态数据源和自检；人物专属内容由项目清单生成。
11. 改写 `package_project.ps1`，移除固定人物名、两套固定皮肤 ID、历史 EXE 名和临时剪贴板 UUID，确保新 idol 项目只依赖 manifest。

完成这些改进后，理想入口可以是：

```powershell
idol-character new --manifest .\idol-project.yaml
idol-character research
idol-character build-anchor
idol-character build-actions
idol-character qa
idol-character package
idol-character release --draft
```

其中 `build-anchor`、动作联系表和最终 `release` 之前必须暂停等待人工批准。自动化的目标不是取消审美与权利判断，而是让目录、协议、构建、检查和发布不再重复手工劳动。
