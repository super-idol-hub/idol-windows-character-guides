# 男性偶像 Windows 角色生成与开发规范

> 男性角色版，由[女性偶像 Windows 角色生成与开发规范](女性偶像Windows角色生成与开发规范.md)及“小曦薇”v3.0.4 的真实运行时经验派生。两版使用相同的高清帧、动画、皮肤、QA 和独立 EXE 协议；本版只重新设计默认表演动作、人物气质配置和男性服装常见的视觉检查项。

本文是一套可执行的半自动 SOP，不代表仓库已经存在“一条命令生成任意男艺人”的成品工具，也不宣称已经制作出某位男性 idol 的 EXE。实际项目仍需要合法或公开可见的参考素材、身份锚点、分行动作生成、人工确认和私有构建环境。

> **文档状态**：这是尚未针对任何具体男性 idol 完成素材、EXE 或 Release 的派生设计规范。动作表是推荐默认值，第一次实际应用时必须经过人物气质审核、逐行动画 QA 和真实 Windows 运行验证；女性外部案例的质量数据不能替代男性项目的实测结果。

## 1. 设计原则

男性版不是把人物统一做成僵硬、强壮、冷脸或“耍帅”。动作选择优先级必须是：

1. 用户明确提出的动作和造型要求。
2. 艺人的公开气质、活动习惯和粉丝熟悉的互动方式。
3. 当前皮肤对应的时代、职业、角色和服装结构。
4. 最后才使用本文的“清爽亲和型男性偶像”默认动作组。

男性角色可以可爱、害羞、落泪、飞吻或跳手势舞；女性角色也可以沉稳、利落或酷感。性别文档只是默认动作起点，不是限制。

默认产品边界：

- Windows 写实或半写实全身桌面互动角色。
- 内置主皮肤可单 EXE 独立启动，不依赖 Codex、Python 或 OpenCV。
- 可选皮肤使用 EXE 同级的 `skins` 文件夹。
- 粉丝个人欣赏、交流、非官方、非商业。
- 不把原始照片、节目视频、歌曲、台词、字幕、Cookie 或账号凭据打进程序。
- 先确认脸和体型，再批量生成动作；主视觉未批准时禁止生成 176 帧。

## 2. 完整流程

```mermaid
flowchart LR
    A[输入男艺人与目标造型] --> B[素材来源与权利边界]
    B --> C[身份锚点与真实体型]
    C --> D[选择人物气质档案]
    D --> E[编写动作脚本]
    E --> F[整行动作生成]
    F --> G[透明化与统一注册]
    G --> H[24 行 176 帧]
    H --> I[运动网格与 24 阶段补间]
    I --> J[交互、皮肤与换装]
    J --> K[自动 QA 与人工审核]
    K --> L[独立 EXE 与发布包]
```

每一步都有人工门禁：

- 主视觉门禁：脸、体型、发型、服装、鞋均通过。
- 动作门禁：动作符合艺人气质，没有不自然的肢体或冒犯性表达。
- 方向门禁：16 个鼠标视线方向通过盲测且不缩放人物。
- 发布门禁：EXE 自检、窗口烟测、免责声明和隐私扫描均通过。

## 4. 参考素材与身份锚点

一个可靠的主皮肤建议至少准备：

- 正面或接近正面的清晰面部 2–4 张。
- 左右三分之四角度各 1–2 张。
- 微笑、大笑、严肃、惊讶或低落等表情 2–4 张。
- 正面全身 1–2 张，必须能看到腿、鞋和真实站姿。
- 同套造型半身或细节图 2–4 张。
- 发型、领口、袖口、腰线、裤型、鞋和配饰局部参考。

优先使用同一时期、同一妆发、同一服装的素材。跨时期素材会让生成模型平均五官、发型和体型。

身份锚点必须锁定：

- 脸型、眼距、眉眼、鼻唇关系、下颌和笑容。
- 真实肩宽、腰线、腿长和头身比。
- 发型分缝、刘海、鬓角、长度、发色和发丝走向。
- 服装领口、袖长、衣摆、裤腰、裤脚、鞋型和配饰。
- 正面自然站姿、双脚完整、手指自然。

不要为了表现“男性”而擅自加宽肩膀、加粗下颌、增加肌肉或放大手脚。目标是还原具体人物，而不是生成泛化的男性模特。

每张素材在私有来源清单中记录文件名、URL、获得日期、用途和是否可公开。公开 Git 仓库不保存原图。

## 5. 人物气质档案

选定一个主档案，再按用户要求覆盖单个动作：

| 档案 | 视觉节奏 | 表情与动作建议 | 避免 |
|---|---|---|---|
| 清爽亲和 | 轻快但不夸张 | 自然微笑、点头、短挥手、比心 | 过度端着或全程冷脸 |
| 沉稳克制 | 幅度小、停顿明确 | 整理袖口、轻点头、抱臂、思考 | 每个动作都叉腰或皱眉 |
| 元气舞台 | 节奏明显、镜头感强 | 节拍手势、舞台律动、应援 pose | 用未经验证的完整走跑步态 |
| 俏皮可爱 | 表情丰富、互动直接 | 鼓脸、害羞、飞吻、假装生气 | 因“男性”而删除可爱动作 |
| 戏剧角色 | 服装和身份驱动 | 拂袖、回眸、职业动作、角色礼仪 | 把现代手势硬放进古装角色 |

同一艺人的不同皮肤可以有不同档案。主皮肤使用公共默认动作，外置角色皮肤通过第 15 行增加专属动作。

## 6. 男性版 24 行、176 帧动作协议

成品使用固定 `528×808` RGBA 关键帧，路径为 `rNN/cNN.png`。24 行共 176 个归档帧：

| 行 | 帧数 | 男性版默认语义 | 动作与运行说明 |
|---:|---:|---|---|
| 0 | 7 个归档槽位 | 安静呼吸、眨眼 | 运行时循环 6 个关键帧，`c06=c00`；不主动转头 |
| 1 | 8 | 旧版向右移动 | 只作兼容，拖拽和随机待机不启用 |
| 2 | 8 | 旧版向左移动 | 只作兼容；镜像前必须确认服装和配饰允许 |
| 3 | 4 | 抬手致意 | 短挥手、点头或轻敬礼三选一 |
| 4 | 5 | 受惊后退 | 肩部一震、睁眼、上身后仰；拖拽期间保持中段 |
| 5 | 8 | 疲惫、无奈 | 肩膀下沉、扶额或轻叹气、整理状态、点头恢复 |
| 6 | 6 | 等待回应 | 轻微前倾、期待目光、双手自然放于身前 |
| 7 | 6 | 认真投入 | 专注、确认、完成后点头；不凭空增加键盘或 UI |
| 8 | 6 | 思考、检查 | 目光轻扫、触下巴或自然托腮、得到结论后恢复 |
| 9 | 8 | 看鼠标方向 0–7 | 眼睛先动，头颈轻微跟随，脚底和身体尺度固定 |
| 10 | 8 | 看鼠标方向 8–15 | 与第 9 行组成连续 16 方向环 |
| 11 | 6 | 拖拽后的不满 | 皱眉、整理衣领/袖口/衣摆、抱臂或单手叉腰、放松 |
| 12 | 8 | 内置皮肤“镜头魅力” | 整理袖口/领口/发型、轻侧身、挑眉或微笑定格 |
| 13 | 8 | 内置皮肤“开怀大笑” | 笑意扩大、肩部轻动、略仰头、呼气平复 |

| 15 | 8 | 外置皮肤专属动作 | 西装整理领带、古装拂袖、运动装热身等，由 `exclusiveAction` 命名 |

| 17 | 8 | 保留 | 兼容位，不改变语义 |

## 7. 变化动作的详细脚本

### 7.1 拖拽：受惊 → 整理衣服 → 克制不满

继续使用已经验证过的反应式拖拽，不恢复走路、慢跑或快跑：

```text
按下但未超过阈值
  → 仍按普通点击处理

移动超过缩放后的拖拽阈值
  → r04 c00-c02：肩部一震、睁眼、上身后仰
  → r04 c03：拖拽期间保持受惊
  → 窗口随鼠标移动，人物本身不迈步

松开
  → r11 c00：皱眉看向鼠标
  → c01-c02：整理被拖乱的领口、袖口或衣摆
  → c03-c04：抱臂、单手叉腰或克制不满
  → c05：放松并回到普通待机
```

不同档案可覆盖：

- 沉稳型：皱眉、整理袖口、抱臂。
- 亲和型：短暂吓到、无奈笑、整理衣服。
- 元气型：允许轻跺一次脚，但不连续踢腿。
- 俏皮型：指一下鼠标、鼓脸或假装生气。
- 古装角色：整理衣袖、拂袖回身，不使用现代叉腰动作。

拖拽最小阈值建议从 8 像素起，并按显示比例调整。普通点击、双击、睡眠唤醒和手机退出必须独立判定。

### 7.2 第 12 行：镜头魅力

推荐 8 个关键姿势：

1. 正面发现镜头。
2. 目光集中，手靠近袖口或领口。
3. 整理袖口、领口或发型。
4. 身体轻微侧转。
5. 头部回正，眼睛看镜头。
6. 微笑或轻挑眉。
7. 定格 pose。
8. 收手并回到站姿。

不要把“镜头魅力”固定为西装整理领带；休闲装、舞台装和古装应使用与服装相符的接触点。

### 7.3 第 13 行：开怀大笑

笑意出现 → 笑容扩大 → 肩部随笑轻动 → 略微仰头 → 最明显的大笑 → 呼气平复 → 保留笑意 → 恢复。

大笑时最容易发生身份漂移。牙齿、嘴角、眼型、下颌和发际线必须与身份锚点一致，不能因为张嘴就换成另一张脸。

### 7.4 第 14 行：低落或落泪

笑意消失 → 低头 → 肩膀下沉 → 湿润眨眼 → 手背轻擦眼角或停顿 → 深呼吸 → 自我安慰地点头 → 恢复。

男性版不禁止哭。如果人物气质不适合落泪，可以把动作命名为“低落一下”或“安静叹气”，但仍保留清晰的情绪变化。

泪滴不要作为与人物分离的漂浮物画进帧中；需要泪光时应贴合眼部轮廓，并确保透明背景清理稳定。

### 7.5 第 18 行：节奏手势

双手准备 → 左右节拍交替 → 胸前节奏手势 → 方向交换 → 动作高潮 → 收势 → 镜头微笑 → 恢复。

双手左右顺序必须明确。手臂不应在补间中穿过胸口；如果手的位置跨越较大，必须增加真正的中间关键姿势。

### 7.6 第 19 行：舞台表演

进入舞台状态 → 上身随节拍轻摆 → 左右重心各一次 → 面向观众互动 → 定格 → 平复 → 收势 → 恢复。

- 不内置音乐、歌词或字幕。
- 麦克风只在目标造型本身拥有并得到批准时出现。
- 只做轻量左右律动，不用运动网格伪造完整走位或复杂舞步。
- 不完整复刻某支受保护的舞蹈编排。

### 7.7 第 20 行：角色演绎

观察 → 转身或回眸 → 衣摆/袖口跟随 → 情绪增强 → 角色定势 → 情绪缓和 → 转回正面 → 恢复。

这一行替代女性样板中的“女侠动作”名称，更适合西装角色、古装角色、校园角色、运动角色或职业角色。动作由皮肤设定决定，不按性别决定。

### 7.8 第 21 行：镜头互动与换肤结束 pose

发现镜头 → 抬手 → 比心、点赞、轻敬礼或飞吻中选择一个 → 微笑定格 → 小幅换 pose → 收手 → 正面站稳 → 最终 pose。

最后一帧必须正面、居中、脚底稳定、脸无遮挡，因为换肤过渡会把它作为新皮肤最终定格。

### 7.9 第 22、23 行：持久状态

坐下看手机：

- `c00-c03`：屈膝、坐稳、取出手机。
- `c03-c05`：低头浏览、轻滑动、偶尔抬眼，持续循环。
- `c06-c07`：收起手机并起身。
- 进入循环后只能由点击请求退出，退出要等到准确关键帧边界。

侧卧睡眠：

- `c00-c04`：蹲下、坐下、手臂支撑、侧卧、闭眼。
- `c04`：持续睡眠。
- `c05-c07`：睁眼、撑起身体、站回待机。
- `z/Z` 由运行时从当前人物头部上方发射，不画进 PNG，也不使用固定窗口坐标。

## 8. 关键帧生成与视觉注册

每个动作通常创作 4–8 张真正不同的关键姿势。生成要求：

1. 每行动作都引用同一身份锚点、全身锚点和目标服装参考。
2. 一整行动作作为一个连续表演生成，避免逐帧独立生成造成换脸。
3. 背景使用与服装不冲突的纯色键，不强制固定绿色或青色。
4. 清除背景、半透明脏边和完全透明像素中的非零 RGB。
5. 同一动作整行共用 viewport、缩放、脚底线和人体中心。
6. 禁止按每一帧自己的包围框自动铺满画布。
7. 大幅手臂、外套、衣摆或身体方向变化必须有真实中间姿势。

男性常见服装专项注意：

- 西装、夹克、长外套和古装袍服转身时轮廓会扩大，必须预留透明边距。
- 手插口袋、整理领口和扶额动作容易让手掌错误消失，要逐帧检查手腕和衣物接触关系。
- 上身侧转会造成肩宽视觉变化，但人物高度、骨盆中心和脚底线不能随之缩放。
- 长袖、腰带、项链、耳饰、眼镜和外套开合不能跨帧消失、换侧或变色。
- 裤型和鞋比裙摆更容易暴露双腿错误；站姿帧要检查两条腿的长度、膝盖方向和鞋底接地。

禁止速度线、地面阴影、光圈、漂浮标点、歌词、标签和与人物分离的特效。

## 9. 16 方向鼠标视线

视线协议与女性版完全相同：

```text
row 9:  000, 022.5, 045, 067.5, 090, 112.5, 135, 157.5
row 10: 180, 202.5, 225, 247.5, 270, 292.5, 315, 337.5
```

- `000°` 是向上，不是正面待机。
- `090°` 是屏幕右，`180°` 是向下，`270°` 是屏幕左。
- 先一起生成并批准上、右、下、左四个基准方向。
- 第 9 行整行生成并注册通过后，才能生成第 10 行。
- 眼睛先移动，眼睑和眉毛配合，头颈和上身只轻微跟随。
- 不通过缩放、整个人旋转、倾斜画布或拉伸头骨制造方向感。
- 两行与普通待机共用人物高度、下半身中心和脚底线。

运行时应有进入/退出迟滞和中心死区。鼠标离开近距离范围后立即回到安静呼吸眨眼，不持续盯着全屏鼠标。

方向 QA 必须包括：四向基准确认、16 方向标注联系表、相邻连续性、三份隔离的盲测和正常桌面尺寸检查。错误方向要重做完整所在行，不能单独拼贴一个头部。

## 10. 高清帧与运动补间

独立 Windows 产品使用：

- 24 行、每行最多 8 列。
- 共 176 个归档 PNG。
- 每帧 `528×808` RGBA。
- 运行时逻辑尺寸 `132×202`。
- 双向 `17×25` 运动网格。
- 普通动作每个完整循环至少 24 个显示阶段。

建议映射：

- 8 个关键姿势 × 每段 3 阶 = 24 阶。
- 6 个关键姿势 × 每段 4 阶 = 24 阶。
- 5 个关键姿势 × 每段 5 阶 = 25 阶。
- 4 个关键姿势 × 每段 6 阶 = 24 阶。

关键原则：

- 使用 smoothstep 缓动并保留动作总时长。
- 中间帧始终只有一个不透明人物轮廓。
- 禁止把前后两张完整人物图做 alpha 交叉淡化。
- 运动网格不能修复错误的人体结构或错误的动作脚本。
- 官方发行的内置和外置皮肤应提供完整运动网格并通过 QA。
- 第三方皮肤 API 最低兼容包可以只有 176 张 PNG；缺少网格时运行时走单姿态回退。

普通人物动作不需要 56 阶。睡眠 `z/Z` 和约 1.2 秒换肤时间线可以独立使用更多轻量阶段。

## 11. 交互状态机

建议优先级：

```text
暂停
  → 换肤
  → 拖拽反应
  → 持续睡眠
  → 持续坐姿
  → 鼠标注视
  → 长关键帧停留
  → 普通动作补间
```

主要状态：

- `Idle`：安静呼吸眨眼。
- `MouseLook`：仅近距离启用。
- `Greeting`、`Waiting`、`Working`、`Review`：一次性动作。
- `DragStartled`：受惊进入和拖拽保持。
- `Displeased`：整理衣服和不满反应。
- `SittingPhone`：进入、持续循环、点击退出。
- `SideRest`：入睡、持续睡眠、点击苏醒。
- `SkinTransition`：候选皮肤验证后锁定输入并原子切换。
- `Paused`：暂停动画但保留安全退出。

普通随机待机以眨眼为主，特殊动作不应连续触发。可以沿用普通空闲 12–25 秒、特殊动作后 18–45 秒的范围，再按人物气质调整。

## 12. 皮肤与专属动作

外置皮肤接口：

```text
<角色>.exe
skins/
  <skin-id>/
    skin.xml
    frames.zip
```

```xml
<?xml version="1.0" encoding="utf-8"?>
<skin apiVersion="1"
      id="formal-suit"
      name="正式西装"
      developer="<开发者>"
      archive="frames.zip"
      exclusiveAction="整理领带致意" />
```

接口要求：

- 文件夹名与 `id` 一致，ID 只允许 `[A-Za-z0-9._-]`。
- `apiVersion="1"`，归档名为 `frames.zip`。
- 文本字段最长 64 字符。
- ZIP 根目录直接包含 `r00/c00.png`，可选网格位于 `motion/`。
- 176 个归档 PNG 必须完整，每张 `528×808` RGBA。
- 单条目不超过 32 MiB；ZIP 和总解压大小各不超过 128 MiB。
- 拒绝绝对路径、`..`、反斜杠、重复条目和未知文件。
- 无效皮肤不影响内置皮肤启动。
- 候选皮肤先离屏验证和预热，换肤完成后原子交换。

第 15 行专属动作按三层选择：

1. 角色层：时代礼仪、职业动作、角色身份。
2. 服装层：西装整理领带、古装拂袖、运动装赛前热身。
3. 艺人层：公开活动中常见但不完整复制具体编舞的招呼或定格 pose。

不能把所有男性皮肤都做成统一“耍帅”。专属动作需要能自然回到第 0 行，道具也必须属于已批准造型。

换肤动画保持：人物 alpha 轮廓分层伪 3D 转身 → 从头到脚替换造型 → 第 21 行末帧定格 → 回到待机。不能翻转整个透明窗口或矩形画布。

## 13. 自动 QA

### 13.1 帧协议

- 24 行、176 个归档 PNG 完整。
- 每张为 `528×808` RGBA 且非空。
- `r00/c06` 与 `r00/c00` 为兼容副本。
- 没有画布边缘裁切、透明 RGB 污染或色键残留。
- 人物高度、骨盆中心、脚底线和头部区域稳定。
- 第 9、10 行与待机比例一致。

### 13.2 动作与运动

- 官方发行皮肤的运动网格完整并可解析。
- 每个普通动作达到至少 24 个显示阶段。
- 中间帧只有一个人物轮廓，没有透明双人重影。
- 大幅动作有真实中间关键姿势。
- 手、腿、外套、袖口和配饰没有断裂或跨帧换侧。
- 第 21 行末帧适合作为换肤结束 pose。

### 13.3 皮肤与安全

- `skin.xml`、ID、文件夹、ZIP 名称和专属动作正确。
- ZIP 没有路径穿越、重复条目、额外文件或解压炸弹风险。
- 损坏皮肤会被跳过并回退内置皮肤。
- 网络功能不使用 Cookie、Token 或登录账号。

### 13.4 运行时

- EXE 自检成功并解码 176 帧。
- 真实透明窗口打开并按时退出。
- 单实例、缩放、拖拽、双击、暂停、托盘和退出有效。
- 手机、睡眠、换肤和不满动作可以完整结束。
- 没有 Codex、Python、OpenCV 和旁置业务 DLL 时仍能启动；Windows 仍需系统 .NET Framework 4.x 组件。
- 断网时可选动态提醒静默失败。

## 14. 男性角色专项人工 QA

自动指标不能判断人物是否像本人。必须查看透明联系表、动作 GIF、深浅背景预览、16 方向视线、换肤录像和 125%/225%/400% 实际桌面截图。

逐项确认：

1. 第一眼能否辨认具体艺人，而不是泛化男模？
2. 所有动作是否保持同一脸型、眉眼、下颌、发际线和发色？
3. 大笑时牙齿和嘴型是否自然，低落时是否突然换脸？
4. 肩宽变化来自真实侧转，还是人物被错误缩放？
5. 整理领口、袖口、领带或口袋时，手腕和衣物接触是否真实？
6. 两条腿、裤脚和鞋是否完整，脚底是否跳动？
7. 外套、长袖、腰带、项链和配饰是否跨帧消失或换侧？
8. 节奏手势中左右手顺序是否清楚，手臂是否穿过身体？
9. 舞台表演是否只是轻量律动，而不是错误的走跑或完整舞蹈复制？
10. 手机是否固定大小、颜色和持握手，循环中是否遮住整张脸？
11. 入睡是否经过蹲下、坐下和支撑，还是从站立突然跳成侧卧？
12. 鼠标移远后是否立即停止注视并回到安静眨眼？
13. 换肤是否只转人物轮廓，末帧是否正面稳定？
14. 可爱、落泪或飞吻动作是否符合该艺人的真实气质，而不是因为性别被机械删除或强加？

## 15. 构建、打包与发布

通用构建链：

1. 从批准的高清动作条带恢复 176 个归档帧。
2. 生成相邻动作和回待机的双向运动网格。
3. 把内置帧 ZIP、图标和 manifest 嵌入 C# WinForms EXE。
4. 编译后显式执行 `--self-test` 和 `--qa-window`。
5. 验证外置皮肤和损坏皮肤回退。
6. 从干净目录解压完整包并烟测。
7. 生成 EXE、皮肤和完整包的 SHA-256。
8. 扫描 Git 暂存区中的原图、绝对路径、凭据和大型制品。
9. 获得明确发布授权后创建 Git 提交、tag 和 GitHub Release。

公开源码仓库只保存运行时代码、构建脚本、文档和皮肤模板。用户参考、生成条带、动作帧、QA 缓存、EXE 和 ZIP 默认不提交 Git。

## 17. 常见故障与修复

| 现象 | 根因 | 修复 |
|---|---|---|
| 像泛化男模，不像本人 | 为表现男性擅自改变肩宽、下颌或肌肉 | 回到真实身份锚点，锁定具体五官和体型 |
| 大笑或低落时换脸 | 表情参考不足，逐帧独立生成 | 补同一人物表情参考，整行动作重做 |
| 整理领口时手消失 | 手和衣物接触关系不清 | 增加接触中间姿势，检查手腕进入路径 |
| 侧身时人物缩小 | 按单帧包围框自动 fit | 整行共用尺度、骨盆中心和脚底线 |
| 外套或项链换侧 | 独立帧身份与服装连续性失败 | 用同一批次整行生成，修最小失败行 |
| 节奏手势手臂穿胸 | 关键姿势跨度过大 | 增加真实中间关键姿势，不依赖光流修人体 |
| 舞台律动像一条腿动 | 错误步态被当成补间输入 | 取消移动步态，改为上身和轻量重心律动 |
| 56 阶段仍卡顿 | 全人物阶段过多，缓存和渲染压力高 | 普通循环使用至少 24 阶并保持总时长 |
| 出现两个人影 | 前后整图 alpha 交叉淡化 | 单一人物轮廓所有权 + 运动网格形变 |
| 看鼠标时变小 | 方向行注册尺度与待机不同 | 第 9、10 行复用待机尺度、脚底和中心 |
| 永久盯着鼠标 | 没有范围、死区和退出迟滞 | 只在近距启用，移远后回普通待机 |
| 睡觉突然倒下 | 缺少蹲下、坐下、支撑过渡 | 重写第 23 行进入段，增加真实关键姿势 |
| 换肤像卡片翻转 | 整个窗口被旋转 | 只切片人物 alpha 轮廓并原子换肤 |

修复遵循最小范围原则：某行动作失败只修该行；方向失败修第 9 或 10 行；皮肤失败只修该皮肤。只有身份锚点本身错误时才重建全套。

## 18. 完成定义

男性 idol Windows 角色只有同时满足以下条件才算完成：

- 用户明确批准主脸、真实体型、发型、服装、鞋和配饰。
- 气质档案和动作脚本得到批准，没有性别刻板化或冒犯性表达。
- 24 行、176 个归档帧完整，所有皮肤使用同一协议。
- 普通动作至少 24 个显示阶段，无双人重影和明显卡顿。
- 拖拽是受惊和不满反应，不依赖错误的走跑步态。
- 手机和睡眠为持久状态，点击后播放完整退出段。
- `z/Z` 从当前人物头部上方发射。
- 16 方向视线只在近距工作，移远后恢复安静眨眼。
- 第 21 行末帧适合作为换肤结束 pose。
- 换肤只处理人物轮廓，候选皮肤验证后原子切换。
- 外套、裤腿、鞋、手、配饰和发型跨帧连续。
- 皮肤损坏或网络断开不会阻止主程序使用。
- EXE 在没有 Codex、Python 和 OpenCV 的 Windows 环境可独立启动。
- 自检、窗口烟测、哈希、隐私扫描和干净环境烟测通过。
- README、版本、免责声明、Release Notes 和程序集元数据一致。
- 公开仓库不含原始参考、私有路径、账号凭据和大型生成制品。
- 公开发布前确认非官方、非商业声明和权利方处理渠道。

## 19. 与女性样板的关系

本男性版没有改变以下经过“小曦薇”验证的核心结论：

- 相似度必须在批量动作之前确认。
- 高清成品必须从高清动作源恢复，不能直接放大小图集。
- 6 张关键姿势过于跳跃时，应增加必要姿势并做 24 阶段补间。
- 56 阶段不是越多越好；普通人物动作 24 阶更平衡。
- 透明交叉淡化会产生双人重影，应使用单一人物轮廓形变。
- 走跑步态不自然时应删除交互，而不是用插帧掩盖。
- 看鼠标方向必须共享待机尺度和脚底线，并有近距退出范围。
- 手机、睡眠和换肤必须是明确状态机，而不是一次性 GIF。

变化只发生在表演语义：女性样板的卖萌、女侠、飞吻和跺脚默认动作，在男性版中改为可配置的镜头魅力、角色演绎、镜头互动和整理衣服后的不满。任何动作仍可根据具体艺人气质恢复、替换或混合。
